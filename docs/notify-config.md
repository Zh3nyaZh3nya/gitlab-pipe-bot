# Конфигурация уведомлений (`<NAME>_CONFIG_JSON`)

Описывает бизнес-логику уведомлений о пайплайне для одной или нескольких
GitLab-репозиториев: список этих репозиториев, ссылку на Jira и список
чатов, в которые нужно слать сообщения, — для каждого чата отдельно
задаются свои теги (разработчики/менеджеры) и фильтр по обязательным джобам.

Бот обслуживает произвольное число конфигов одновременно, каждый — в своей
переменной окружения.

## Расположение и настройка

Конфиг задается не файлом, а переменной окружения вида `<NAME>_CONFIG_JSON`
— JSON-строкой, которую в рантайме парсит `src/notify.config.ts`. При старте
`loadNotifyConfigs()` сканирует `process.env` и забирает значения всех
переменных, чье имя оканчивается на `_CONFIG_JSON`, — например
`PROJECT_CONFIG_JSON`. `<NAME>` — это просто человекочитаемая метка
переменной (обычно название проекта/группы, для которого она заведена);
на сопоставление с входящими webhook-событиями он не влияет — за это
отвечает поле `repositories` внутри самого JSON (см. ниже).

Переменных может быть сколько угодно, например:

```
PROJECT_CONFIG_JSON='{ "repositories": ["group/project-backend", "group/project-frontend"], ... }'
SOME_OTHER_PROJECT_CONFIG_JSON='{ "repositories": ["group/some-other-repo"], ... }'
```

Чтобы добавить конфиг:

1. Добавьте в `.env` (см. `.env.example`) переменную `<NAME>_CONFIG_JSON` со
   значением в одинарных кавычках, содержащим JSON вида `NotifyConfig`,
   включая `repositories` — список путей GitLab-репозиториев
   (`namespace/repo`), к которым этот конфиг относится (см. «Как
   связывается репозиторий» ниже).
2. Тип описан в `src/types/notify-config.ts` и импортируется в
   `src/pipeline.ts`.

Если на старте не нашлось ни одной переменной, оканчивающейся на
`_CONFIG_JSON`, приложение упадет с ошибкой
`Missing required env variable: нету переменной *_CONFIG_JSON`.

В проде переменные нужно задавать там же, где заданы `BOT_TOKEN`/
`GITLAB_WEBHOOK_SECRET` (Deployment/Secret в кластере) — этот репозиторий
манифесты Deployment не хранит.

## Как связывается репозиторий

Все GitLab-репозитории шлют webhook на один и тот же URL/секрет — конкретный
конфиг для входящего pipeline-события ищется по пути репозитория:

1. `getNotifyConfigForProject` (в `src/notify.config.ts`) берет
   `project.path_with_namespace` из тела webhook-события (например
   `"group/project-front"` — включает namespace/группу, а не только
   короткое имя проекта).
2. Среди всех загруженных конфигов ищется тот, чей `repositories` содержит
   этот путь (сравнение без учета регистра).
3. Если совпадение не найдено, `src/index.ts` пропускает событие и отвечает
   `200 { skipped: "нет конфига для ..." }`, ничего не отправляя в Telegram.

Поэтому `project.path_with_namespace` пришедшего события должен дословно
(без учета регистра) присутствовать в чьем-то `repositories` — иначе
уведомления по этому репозиторию отправляться не будут.

## Как получить `chat_id`

1. Напишите боту в личку `/start`, либо добавьте бота в нужную группу/канал и отправьте туда любое сообщение.
2. `https://api.telegram.org/botBOT_TOKEN/getUpdates`
3. В ответе найдите `"chat":{"id": ... }`:
   - для личного чата — положительное число;
   - для группы/супергруппы — отрицательное, возможно будет начинаться с `-100`.
4. Скопируйте это число в `chat_id` соответствующего элемента `chats`.

Если для группы `getUpdates` возвращает пустой список (`"result":[]`) — у
бота включен privacy mode, и он не видит обычные сообщения в группе.
Отправьте боту в BotFather команду `/setprivacy` → выберите бота →
`Disable`, затем повторите шаги 1–2.

## Поля

### `repositories: string[]`

Список путей GitLab-репозиториев (`project.path_with_namespace` из
webhook-события, например `"group/project-front"`), к которым применяется
этот конфиг — см. «Как связывается репозиторий» выше. Один репозиторий не
может быть в двух конфигах одновременно (используется первое совпадение).

### `linkToJira: string | null`

Базовая ссылка на доску Jira, общая для всех чатов, например:

```
https://example.atlassian.net/jira/software/projects/name_project/boards/3?jql=
```

Логика вставки ссылки в сообщение (`buildBranchLink` в `src/pipeline.ts`):

1. Из `linkToJira` вырезается ключ проекта — сегмент пути сразу после
   `projects/`, то есть `{key}` в `.../projects/{key}/{tab}/...`.
2. Если ключ найден **и** название ветки пайплайна (`pipeline.ref`)
   содержит этот ключ (например ветка `feature/NP-123-fix`), в сообщение
   вставляется кликабельная ссылка:
   ```
   {linkToJira}&selectedIssue={ref}
   ```
   (используется `&`, если в `linkToJira` уже есть `?`, иначе `?`).
3. Если `linkToJira === null`, ключ не найден, либо ветка его не содержит —
   в сообщение подставляется обычное экранированное название ветки без ссылки.

### `chats: NotifyChatConfig[]`

Список чатов-получателей. Уведомление о пайплайне рассылается **отдельно в
каждый чат** из списка — сообщение и статус (успех/провал) редактируются
для каждого чата независимо, свой фильтр по джобам может пропустить
уведомление в одни чаты и пропустить в другие.

```ts
{
  chat_id: number;
  userNames: {
    developers: { gitlab_name: string; telegram_name: string }[] | null;
    managers: { telegram_name: string }[] | null;
  };
  targetBuilds: string | null;
}
```

- **`chat_id`** — id чата/группы, куда отправляется уведомление (см. «Как
  получить `chat_id`» выше).
- **`userNames.developers`** — сопоставление gitlab-логина (`user.username`
  из webhook-события) с telegram-ником. Используется, чтобы **тегнуть
  разработчика, если пайплайн завершился со статусом `failed`**: в конце
  сообщения добавляется `@telegram_name` того, кто запушил коммит. Если
  автор не найден в списке, тег не добавляется — сообщение отправляется как
  обычно. `null` (или пустой список) — тегать некого, тег не добавляется.
- **`userNames.managers`** — список telegram-ников, которых нужно **тегнуть
  при успешном пайплайне** (`status: success`). Тегаются все менеджеры из
  списка сразу, через пробел. `null` (или пустой список) — тегать некого,
  тег не добавляется.
- **`targetBuilds`** — список имен джоб через запятую, например
  `"bake-prod,deploy-prod"`. Уведомление в этот чат отправляется, только
  если **все** перечисленные джобы присутствуют среди `builds` пришедшего
  пайплайна (`buildNames.includes(...)` для каждого имени). Если какой-то
  из джоб нет — уведомление в этот чат не отправляется (остальные чаты из
  списка это не затрагивает). `null` — фильтр по джобам отключен,
  уведомление отправляется для любого набора джоб (при условии, что статус
  пайплайна вообще поддерживается, см. `STATUS_LABELS` в `src/pipeline.ts`).

## Пример

См. `.env.example` для минимального значения. Развернутый пример на два
чата для репозиториев проекта `kupipolis` (переменная
`KUPIPOLIS_CONFIG_JSON`, значение в `.env` должно быть в одну строку или в
кавычках, как в `.env.example`):

```json
{
	"repositories": ["group/project-backend", "group/project-frontend"],
	"linkToJira": "https://example.atlassian.net/jira/software/projects/NP/boards/3?jql=",
	"chats": [
		{
			"chat_id": -1001234567890,
			"userNames": {
				"developers": [
					{ "gitlab_name": "ivan.petrov", "telegram_name": "ivan_petrov_tg" },
					{ "gitlab_name": "anna.sidorova", "telegram_name": "anna_sidorova_tg" }
				],
				"managers": [
					{ "telegram_name": "project_manager_tg" }
				]
			},
			"targetBuilds": "bake-prod,deploy-prod"
		},
		{
			"chat_id": -1009876543210,
			"userNames": {
				"developers": [],
				"managers": []
			},
			"targetBuilds": null
		}
	]
}
```
